Overview

What is this Project?

This project involves building a fully containerized web application that runs inside Docker containers. Python Flask is used for the backend, PostgreSQL for the database, and both containers are connected via an IPvlan network where each container has its own static IP address. Data is permanently saved using a Named Volume.

Technologies Used
🐍

Python Flask

Lightweight web framework. Two routes created — / serves the webpage and /api returns a JSON response.

🐘

PostgreSQL

Production-grade relational database. A users table is created via init.sql with id and name columns.

🐳

Docker

Multi-stage Dockerfile used — builder stage installs dependencies, runtime stage keeps only what's needed. A non-root user is added for security.

🔧

Docker Compose

Both containers start with a single command — docker compose up --build -d. They start in the correct order automatically via depends_on.

🌐

IPvlan Network

Each container is given a dedicated static IP — backend gets 192.168.170.10, database gets 192.168.170.11. Layer 2 mode is used.

💾

Named Volume

The pgdata named volume ensures PostgreSQL data is permanently saved — even if the container is stopped or deleted, the data remains safe.

Key Concepts

Important Learnings

Why Multi-stage Dockerfile?

Stage 1 (builder) installs all dependencies. Stage 2 (runtime) copies only what is necessary — this results in a smaller and more secure final image. Adding a non-root user further improves security.

IPvlan vs Bridge Network

In a Bridge network, all containers share the same network. In IPvlan, each container gets its own dedicated IP address — like different phones having different numbers. Containers can communicate directly using their IPs.

Why is a Named Volume necessary?

A container is temporary — when it stops, all data inside is lost. A Named Volume is permanent storage — even if the container is removed, the data stays safe. This is the core concept of persistence.

Benefit of Docker Compose

Instead of running multiple docker run commands separately, everything is defined in one file. The entire stack comes up with one command and goes down with one command. depends_on ensures containers start in the correct order.
