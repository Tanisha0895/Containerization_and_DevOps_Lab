Comparison of Docker Run and Docker Compose

← Back to Home

Objective

To understand the relationship between docker run and Docker Compose, and to compare their configuration syntax and use cases.

1. Background Theory

1.1 Docker Run (Imperative Approach)

The docker run command is used to create and start a container from an image. It requires explicit flags for port mapping, volume mounting, environment variables, network configuration, restart policies, resource limits, and container name.

docker run -d \
  --name my-nginx \
  -p 8080:80 \
  -v ./html:/usr/share/nginx/html \
  -e NGINX_HOST=localhost \
  --restart unless-stopped \
  nginx:alpine

1.2 Docker Compose (Declarative Approach)

Docker Compose uses a YAML file to define services, networks, and volumes in a structured format. Instead of multiple docker run commands, a single command is used.

docker compose up -d

3. Practical Tasks

Task 1: Single Container — Docker Run

docker run -d --name lab-nginx -p 8081:80 nginx:alpine

docker ps

docker stop lab-nginx
docker rm lab-nginx

Task 1: Single Container — Docker Compose

services:
  nginx:
    image: nginx:alpine
    container_name: lab-nginx
    ports:
      - "8081:80"
docker compose up -d
docker compose ps
docker compose down

Task 2: Multi-Container — WordPress + MySQL

services:

    restart: always

CMD ["node", "app.js"]

4. Advantages of Docker Compose

  1. Simplifies multi-container applications
  2. Provides reproducibility
  3. Version controllable configuration
  4. Unified lifecycle management
  5. Supports service scaling

5. Conclusion

Docker Run is imperative — you specify step-by-step instructions. Docker Compose is declarative — you define the desired state. Compose is better for multi-container applications as it simplifies management and provides reproducibility.
